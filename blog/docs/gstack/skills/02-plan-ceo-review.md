# `/plan-ceo-review`

> **一句话定位：** CEO 视角的计划审查。重新思考问题本质，找到10倍更好的产品，挑战前提，并在四种模式下对计划进行全面的战略与工程审查。

---

## **概述**

`/plan-ceo-review` 是 gstack 中最具战略视角和前瞻性的计划审查技能。它的核心使命不是形式化审批，而是在项目启动前，深度洞察并暴露潜在风险与机会，使计划具备非凡的可实现性和突破性。

**它不写代码，不修改实现。** 它只做一件事：用最大的严格性和适当的野心审查计划。

**触发时机：**

- 你说"想得更大一点"、"扩展范围"、"战略审查"、"重新思考这个"、"这够有野心吗"
- 你在质疑一个计划的范围或雄心
- 一个计划感觉可以更有远见

**使用顺序：** `/office-hours` → **`/plan-ceo-review`** → `/plan-eng-review` → 开始实现

---

## **四种工作模式**

这是 `/plan-ceo-review` 最核心的设计。运行之前，你需要选择一种模式，技能会完全按照该模式执行，不会悄悄漂移。

### **范围扩展（SCOPE EXPANSION）**

你在建造一座大教堂。描绘柏拉图式的理想状态。向上推动范围。问"什么能让这个产品用2倍的工作量提供10倍的价值？"每一个扩展都单独作为一个问题提出，由你逐一决定是否纳入。

**适用场景：** 全新功能、你说了"大胆做"或"建造大教堂"的时候。

### **选择性扩展（SELECTIVE EXPANSION）**

当前范围是基准，先让它无懈可击，然后把所有可能的扩展机会一一列出，每个单独提问，让你自己挑选。推荐立场中立——只呈现机会、工作量和风险，让你决定。

**适用场景：** 对现有系统的功能增强、你说"保持范围但诱惑我"的时候。

### **保持范围（HOLD SCOPE）**

计划的范围已经确定。任务是让它无懈可击——架构、安全、边界情况、可观测性、部署。不扩展，也不缩减。

**适用场景：** Bug修复、热修复、重构、你明确说"就这样"的时候。

### **范围缩减（SCOPE REDUCTION）**

你是一名外科医生。找到能实现核心目标的绝对最小版本，其他一切全部推迟。毫不留情。

**适用场景：** 计划触及超过15个文件、计划感觉过度构建或方向错误的时候。

## **核心哲学**

### **完整性很便宜**

AI 编程将实现周期从数周压缩到数分钟。在比较“方案A（完整约150行）”和“方案B（精简但仅覆盖90%约80行）”时，应始终选择A——补齐那70行代码对 Claude Code 来说只需片刻。“追求上线速度”源于人类开发受限时代，如今已过时。应毫无保留地实现完整方案，彻底释放AI编程的潜力。

| 任务类型 | 人类团队 | CC+gstack | 压缩比 |
| -------- | -------- | --------- | ------ |
| 样板代码 | 2天      | 15分钟    | ~100x  |
| 测试     | 1天      | 15分钟    | ~50x   |
| 功能     | 1周      | 30分钟    | ~30x   |
| Bug修复  | 4小时    | 15分钟    | ~20x   |

### **用户永远在掌控**

在所有模式下，用户100%掌控。每一个范围变更都需要明确的选择加入——绝不悄悄添加或删除范围。一旦用户选择了模式，就完全遵守。

### **零静默失败**

每一个失败模式都必须可见——对系统、对团队、对用户。如果一个失败可以静默发生，那就是计划中的一个严重缺陷。

## **完整工作流程**

### **系统预审（PRE-REVIEW SYSTEM AUDIT）**

在做任何事情之前，先运行系统审计。这不是计划审查本身，而是智能审查计划所需要的上下文。

技能会运行以下命令收集情报：

- `git log --oneline -30` — 最近的提交历史
- `git diff <基础分支> --stat` — 已经发生了什么变化
- `git stash list` — 是否有暂存的工作
- 搜索代码中的 `TODO/FIXME/HACK/XXX` 注释
- 统计最近30天内频繁被修改的文件

同时读取 `CLAUDE.md`、`TODOS.md` 和任何现有的架构文档。

**设计文档检查：** 如果存在来自 `/office-hours` 的设计文档，读取它作为问题陈述、约束条件和已选方案的事实来源。

**移交记录检查：** 如果存在上一次 CEO 审查会话暂停时留下的移交记录，读取它，避免重复提问。

**如果没有设计文档：** 会询问你是否要先运行 `/office-hours`（推荐），或直接进行标准审查。

### **第0步：核心范围挑战与模式选择**

#### **0A：前提质疑**

1. 这是正确的问题吗？不同的框架能否产生更简单或更有影响力的解决方案？
2. 实际的用户/业务结果是什么？这个计划是达到该结果的最直接路径，还是在解决一个代理问题？
3. 如果什么都不做会怎样？这是真实的痛点还是假设的？

#### **0B：现有代码利用**

1. 哪些现有代码已经部分或完全解决了每个子问题？把每个子问题映射到现有代码。
2. 这个计划是否在重建已经存在的东西？如果是，解释为什么重建比重构更好。

#### **0C：理想状态映射**

描述12个月后这个系统的理想终态。这个计划是在向那个状态迈进还是偏离？

```
当前状态          这个计划          12个月理想状态
[描述]    --->   [描述增量]   --->   [描述目标]
```

#### **0C-bis：实现方案生成（必须执行）**

在选择模式之前，必须提出2-3个不同的实现方案：

```
方案 A：[名称]
摘要：[1-2句话]
工作量：[S/M/L/XL]
风险：[低/中/高]
优点：[2-3条]
缺点：[2-3条]
可复用：[现有代码/模式]

方案 B：[名称]
...

方案 C：[名称]（可选）
...
```

规则：

- 至少2个方案，非简单计划建议3个
- 必须有一个"最小可行"方案（文件最少、改动最小）
- 必须有一个"理想架构"方案（长期最优）
- 在用户批准选定方案之前，不得进入模式选择

#### **0D：模式专属分析**

**范围扩展模式额外执行：**

- **10倍检查：** 用2倍的工作量实现10倍价值的版本是什么？具体描述。
- **柏拉图式理想：** 如果世界上最好的工程师有无限时间和完美品味，这个系统会是什么样子？用户使用时会有什么感受？
- **令人愉悦的机会：** 哪些相邻的30分钟改进会让这个功能大放异彩？至少列出5个。
- **扩展选择仪式：** 先描述愿景，然后把具体的范围提案逐一作为独立问题提出。每个提案有三个选项：A）加入本次计划范围 B）推迟到TODOS.md C）跳过。

**选择性扩展模式额外执行：**

- **复杂度检查：** 如果计划触及超过8个文件或引入超过2个新类/服务，这是一个代码味道。
- 先运行"保持范围"分析，然后扫描扩展机会。
- **逐一摘选仪式：** 把每个扩展机会作为独立问题提出，中立立场，让用户决定。

**保持范围模式额外执行：**

- 复杂度检查：是否可以用更少的活动部件实现相同目标？
- 哪些工作可以推迟而不阻碍核心目标？

**范围缩减模式额外执行：**

- 无情削减：能向用户交付价值的绝对最小版本是什么？
- 什么可以作为后续PR？分离"必须一起发布"和"最好一起发布"。

#### **0D-POST：持久化 CEO 计划（仅扩展模式）**

选择仪式结束后，将愿景和决策写入磁盘，保存到 `~/.gstack/projects/{项目名}/ceo-plans/{日期}-{功能名}.md`，格式包含：

- 状态（ACTIVE）
- 10倍愿景描述
- 柏拉图式理想描述
- 范围决策表（提案、工作量、决策、理由）
- 已接受的范围列表
- 已推迟到TODOS.md的列表

文档写完后，会由独立子代理进行对抗性审查（最多3轮），从完整性、一致性、清晰度、范围、可行性五个维度检查，并报告质量分数。

#### **0E：时间审讯（扩展、选择性扩展、保持范围模式）**

提前思考实现过程中需要做的决策，现在就解决：

```
第1小时（基础）：实现者需要知道什么？
第2-3小时（核心逻辑）：他们会遇到什么模糊之处？
第4-5小时（集成）：什么会让他们感到惊讶？
第6小时以后（打磨/测试）：他们希望提前计划好什么？
```

（注：这些是人类团队的实现小时数。使用 CC+gstack，6小时的人类实现压缩到约30-60分钟。）

#### **0F：模式选择**

技能会呈现四个选项，根据上下文给出默认推荐：

- 全新功能 → 默认"范围扩展"
- 对现有系统的功能增强 → 默认"选择性扩展"
- Bug修复或热修复 → 默认"保持范围"
- 重构 → 默认"保持范围"
- 计划触及超过15个文件 → 建议"范围缩减"

### **10个审查章节**

模式和方案确定后，进入10个结构化审查章节。每个章节独立提问，不批量合并。

#### **第1节：架构审查**

评估并绘制图表：

- 整体系统设计和组件边界（依赖关系图）
- **数据流的四条路径：** 对每个新数据流，ASCII图表展示：快乐路径（数据正确流动）、空值路径（输入为nil/缺失时会怎样）、空路径（输入存在但为空/零长度时会怎样）、错误路径（上游调用失败时会怎样）
- **状态机：** 对每个新的有状态对象进行ASCII图表，包括不可能/无效的转换以及防止它们的机制
- 耦合关注点、扩展特性、单点故障、安全架构、生产失败场景、回滚姿态

**扩展模式额外问：** 什么会让这个架构不仅正确，而且优雅？什么基础设施会让这个功能成为其他功能可以构建的平台？

**必须产出：** 显示新组件及其与现有组件关系的完整系统架构ASCII图。

#### **第2节：错误与恢复映射**

这是捕获静默失败的章节，不可跳过。

对每个可能失败的新方法、服务或代码路径，填写表格：

```
方法/代码路径      | 可能出错的地方          | 异常类
-------------------|------------------------|------------------
ExampleService#call| API超时                | TimeoutError
                   | API返回429             | RateLimitError
                   | API返回格式错误的JSON   | JSONParseError

异常类              | 已处理? | 处理动作           | 用户看到
--------------------|---------|-------------------|----------
TimeoutError        | 是      | 重试2次，然后抛出  | "服务暂时不可用"
JSONParseError      | 否←缺口  | —                 | 500错误←糟糕
```

规则：

- 捕获所有异常（`rescue StandardError`、`except Exception`）**永远是代码味道**，必须命名具体异常
- 每个被捕获的错误必须要么重试、要么优雅降级、要么带上下文重新抛出
- 对于 LLM/AI 服务调用：响应格式错误时怎样？响应为空时怎样？模型返回拒绝时怎样？每种都是独立的失败模式

#### **第3节：安全与威胁模型**

安全不是架构的子项，它有自己的章节。

评估：

- 攻击面扩展（新端点、新参数、新文件路径、新后台任务）
- 输入验证（每个新用户输入：nil、空字符串、类型错误、超长、Unicode边界情况、HTML/脚本注入）
- 授权（每个新数据访问：是否限定了正确的用户/角色？是否存在直接对象引用漏洞？）
- 密钥和凭证（新密钥？在环境变量中，不是硬编码？可轮换？）
- 注入向量（SQL、命令、模板、LLM提示注入）
- 审计日志（敏感操作是否有审计追踪？）

对每个发现：威胁、可能性（高/中/低）、影响（高/中/低）、计划是否减轻了它。

#### **第4节：数据流与交互边界情况**

对每个新数据流，产出ASCII图表：

```
输入 ──▶ 验证 ──▶ 转换 ──▶ 持久化 ──▶ 输出
 │         │        │         │         │
 ▼         ▼        ▼         ▼         ▼
[nil?]  [无效?]  [异常?]   [冲突?]   [过期?]
[空?]   [过长?]  [超时?]   [重复键?] [部分?]
```

对每个新用户可见交互，评估边界情况：

```
交互          | 边界情况              | 已处理? | 如何处理?
-------------|----------------------|---------|--------
表单提交      | 双击提交              | ?       |
             | 使用过期CSRF提交      | ?       |
异步操作      | 用户在操作中途离开    | ?       |
             | 操作超时              | ?       |
列表/表格视图  | 零结果               | ?       |
             | 10,000条结果          | ?       |
后台任务      | 任务在处理10项中的3项 | ?       |
             | 后处理失败            |         |
```

#### **第5节：代码质量审查**

评估：

- 代码组织和模块结构（新代码是否符合现有模式？）
- DRY违规（积极标记，引用文件和行号）
- 命名质量（新类、方法、变量是否按它们做什么来命名，而不是怎么做）
- 过度工程检查（是否有新的抽象在解决一个还不存在的问题？）
- 欠工程检查（是否有脆弱的、只假设快乐路径的、缺少明显防御性检查的东西？）
- 圈复杂度（标记任何分支超过5次的新方法，提出重构建议）

#### **第6节：测试审查**

制作该计划引入的每个新事物的完整图表：

```
新用户体验流程：[列出每个新的用户可见交互]
新数据流：[列出每条新的数据路径]
新代码路径：[列出每个新的分支、条件或执行路径]
新后台任务/异步工作：[列出每个]
新集成/外部调用：[列出每个]
新错误/恢复路径：[列出每个]
```

对图表中的每个项目：

- 什么类型的测试覆盖它？（单元/集成/系统/E2E）
- 计划中是否存在对它的测试？如果没有，写出测试规范头部
- 快乐路径测试是什么？
- 失败路径测试是什么？（具体说明哪种失败）
- 边界情况测试是什么？（nil、空、边界值、并发访问）

**测试雄心检查：**

- 什么测试能让你有信心在周五凌晨2点发布？
- 一个充满敌意的QA工程师会写什么测试来破坏这个？
- 混沌测试是什么？

#### **第7节：性能审查**

评估：

- N+1查询（每个新的关联遍历：是否有includes/preload？）
- 内存使用（每个新数据结构：生产中的最大大小？）
- 数据库索引（每个新查询：是否有索引？）
- 缓存机会（每个昂贵的计算或外部调用：是否应该缓存？）
- 连接池压力（新的DB连接、Redis连接、HTTP连接？）
- 最慢的3条新代码路径和估计的p99延迟

#### **第8节：可观测性与可调试性审查**

评估：

- 日志（每个新代码路径：在入口、出口和每个重要分支有结构化日志行？）
- 指标（每个新功能：什么指标告诉你它在工作？什么告诉你它坏了？）
- 追踪（新的跨服务或跨任务流：是否传播了追踪ID？）
- 告警（应该存在哪些新告警？）
- 仪表板（第1天想要哪些新的仪表板面板？）
- 可调试性（如果在发布3周后报告了一个bug，你能仅从日志中重建发生了什么吗？）
- Runbook（每个新失败模式的操作响应是什么？）

**扩展模式额外问：** 什么可观测性会让这个功能成为运维的乐趣？

#### **第9节：部署与发布审查**

评估：

- 迁移安全性（每个新DB迁移：向后兼容？零停机？表锁？）
- 功能标志（任何部分是否应该在功能标志后面？）
- 发布顺序（正确的顺序：先迁移，再部署？）
- 回滚计划（明确的逐步说明）
- 部署时间风险窗口（旧代码和新代码同时运行时，什么会出错？）
- 发布后验证清单（前5分钟？前1小时？）

**扩展模式额外问：** 什么部署基础设施会让发布这个功能成为常规操作？

#### **第10节：长期轨迹审查**

评估：

- 引入的技术债务（代码债务、运维债务、测试债务、文档债务）
- 路径依赖（这会让未来的变更更难吗？）
- 知识集中（文档是否足以让新工程师上手？）
- 可逆性（1-5分：1=单向门，5=容易可逆）
- 一年后的问题（12个月后作为新工程师阅读这个计划——明显吗？）

**扩展模式额外问：** 这发布后接下来是什么？第2阶段？第3阶段？架构是否支持那个轨迹？这是否创造了其他功能可以利用的能力？

#### **第11节：设计与UX审查（仅当检测到UI范围时）**

CEO 请来设计师。不是像素级审计（那是 `/plan-design-review` 和 `/design-review` 的工作），而是确保计划有设计意图。

评估：

- 信息架构（用户看到什么是第一、第二、第三？）
- 交互状态覆盖图（加载/空/错误/成功/部分）
- 用户旅程连贯性（故事板情感弧线）
- AI 垃圾风险（计划是否描述了通用UI模式？）
- DESIGN.md 一致性（计划是否与声明的设计系统匹配？）
- 响应式意图（移动端是被提及还是事后想到？）
- 可访问性基础（键盘导航、屏幕阅读器、对比度、触摸目标）

**必须产出：** 显示屏幕/状态和转换的用户流程ASCII图。

### **外部声音——独立计划挑战（可选，推荐）**

所有审查章节完成后，提供来自不同AI系统的独立第二意见。两个模型对一个计划达成一致，比一个模型的彻底审查更有说服力。

如果 Codex 可用，调用它进行独立的计划挑战；否则使用 Claude 子代理。提示词要求它：

- 找到逻辑漏洞和未陈述的假设
- 寻找根本性的更简单方法（审查是否陷入了细节中）
- 找到可行性风险
- 找到缺失的依赖或排序问题
- 找到战略错误校准（这是正确的构建对象吗？）

如果外部声音与审查发现有分歧，标注为"跨模型张力"并逐一询问是否加入TODOS.md。

### **必须产出的输出**

审查完成后，技能必须产出以下所有内容：

**"不在范围内"章节** — 列出被考虑并明确推迟的工作，每项一行理由。

**"已存在什么"章节** — 列出部分解决子问题的现有代码/流程以及计划是否复用它们。

**"理想状态差距"章节** — 这个计划完成后，我们距离12个月理想状态还有多远。

**错误与恢复注册表** — 来自第2节的完整表格。

**失败模式注册表：**

```
代码路径 | 失败模式 | 已处理? | 已测试? | 用户看到? | 已记录?
```

任何"已处理=否、已测试=否、用户看到=静默"的行 → **严重缺口**。

**TODOS.md 更新** — 每个潜在TODO作为独立问题提出，包含：做什么、为什么、优点、缺点、上下文、工作量估算、优先级、依赖项。

**必须产出的图表：**

1. 系统架构图
2. 数据流图（包括影子路径）
3. 状态机图
4. 错误流图
5. 部署顺序图
6. 回滚流程图

### **审查就绪仪表板**

审查完成后，技能会读取审查日志并显示仪表板：

```
+====================================================================+
| 审查就绪仪表板                                                       |
+====================================================================+
| 审查          | 次数 | 最后运行          | 状态     | 是否必须 |
|--------------|------|------------------|----------|---------|
| 工程审查      | 1    | 2026-03-16 15:00 | 通过     | 是      |
| CEO审查       | 1    | 2026-03-31 10:00 | 通过     | 否      |
| 设计审查      | 0    | —                | —        | 否      |
| 对抗性审查    | 0    | —                | —        | 否      |
| 外部声音      | 0    | —                | —        | 否      |
+--------------------------------------------------------------------+
| 结论：已通过 — 工程审查通过                                           |
+====================================================================+
```

**审查层级说明：**

- **工程审查（默认必须）：** 唯一阻止发布的审查。覆盖架构、代码质量、测试、性能。
- **CEO审查（可选）：** 适用于重大产品/业务变更、新用户面向功能、范围决策。Bug修复和重构可跳过。
- **设计审查（可选）：** 适用于UI/UX变更。纯后端或基础设施变更可跳过。
- **对抗性审查（自动）：** 根据diff大小自动调整。小diff（<50行）跳过，中等diff（50-199行）进行跨模型对抗，大diff（200+行）进行全部4轮审查。
- **外部声音（可选）：** 来自不同AI模型的独立计划审查。永远不阻止发布。
